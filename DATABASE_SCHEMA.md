# Accountant Office Management System — Database Schema

## 1. Purpose

This document defines the MVP database design for SQLite + SQLAlchemy.

The schema is intentionally small, readable, and suitable for a local internal MVP. It must use SQLAlchemy models and remain reasonably portable to PostgreSQL later.

## 2. Data Rules

- Use integer primary keys in the MVP.
- Use UTC-aware timestamps in application code where practical; display dates/times in the chosen local UI format.
- Use `Decimal` in Python and `Numeric(14, 2)` for money. Never use `float` for money.
- Store VAT rate as `Numeric(5, 2)`, for example `18.00`.
- Store document dates as a single `document_date` field. Month/year reports derive values from this field.
- Validate all enum-like values in Pydantic schemas and, where useful, database constraints.
- `created_at` is set when a row is created; `updated_at` changes whenever a row is changed.
- Never store raw passwords, card data, real API keys, or real sensitive demo data.

## 3. Entity Relationship Overview

```text
User

OfficeSettings

Client 1 ───── * Document
Client 1 ───── * Task
Client 1 ───── * Payment

Document 1 ─── * Task       (task.document_id is optional)
Document 1 ─── * Payment    (payment.document_id is optional)

IntegrationStatus
```

## 4. Tables

### 4.1 `users`

Represents accountant/office staff.

| Field | Type | Rules |
|---|---|---|
| `id` | Integer | Primary key |
| `name` | String(120) | Required |
| `email` | String(255) | Required, normalized to lowercase, unique |
| `password_hash` | String(255) | Required; never return in API responses |
| `role` | String(20) | Required; `admin` or `staff` |
| `is_active` | Boolean | Default `true` |
| `created_at` | DateTime | Required |
| `updated_at` | DateTime | Required |

Indexes/constraints:

- Unique index on `email`.
- Check role against allowed values in schema validation.

### 4.2 `office_settings`

One row for the MVP office configuration.

| Field | Type | Rules |
|---|---|---|
| `id` | Integer | Primary key; seed/use one settings record |
| `accountant_name` | String(120) | Required |
| `office_name` | String(160) | Required |
| `default_vat_rate` | Numeric(5,2) | Required; default `18.00`; must be `>= 0` |
| `default_currency` | String(3) | Required; default `ILS` |
| `allowed_file_extensions` | Text | Required; JSON/text list of approved extensions |
| `created_at` | DateTime | Required |
| `updated_at` | DateTime | Required |

MVP rule: the API returns/updates only the single office settings record. Do not make the UI pretend it manages multiple offices.

### 4.3 `clients`

| Field | Type | Rules |
|---|---|---|
| `id` | Integer | Primary key |
| `client_name` | String(160) | Required |
| `business_name` | String(160) | Optional |
| `phone` | String(40) | Optional |
| `email` | String(255) | Optional, validate format when present |
| `business_id` | String(80) | Optional; trim whitespace |
| `client_type` | String(30) | Required enum |
| `address` | Text | Optional |
| `status` | String(20) | Required; `active` or `inactive` |
| `notes` | Text | Optional, internal only |
| `created_at` | DateTime | Required |
| `updated_at` | DateTime | Required |

Allowed `client_type` values:

```text
private_client
exempt_dealer
authorized_dealer
company
other
```

Suggested indexes:

- `client_name`
- `business_name`
- `status`
- `client_type`
- `business_id`

### 4.4 `documents`

| Field | Type | Rules |
|---|---|---|
| `id` | Integer | Primary key |
| `client_id` | Integer | Required foreign key to `clients.id` |
| `document_name` | String(255) | Required |
| `document_type` | String(30) | Required enum |
| `original_filename` | String(255) | Required when a file is uploaded |
| `stored_filename` | String(255) | Required when a file is uploaded; generated by server |
| `file_path` | String(500) | Required when a file is uploaded; internal/private path only |
| `mime_type` | String(120) | Required when a file is uploaded |
| `file_size_bytes` | Integer | Required when a file is uploaded; must be `>= 0` |
| `document_date` | Date | Required |
| `amount_before_vat` | Numeric(14,2) | Required; must be `>= 0` |
| `vat_rate` | Numeric(5,2) | Required; saved per document |
| `vat_amount` | Numeric(14,2) | Required; calculated value |
| `total_amount` | Numeric(14,2) | Required; calculated value |
| `status` | String(30) | Required enum |
| `notes` | Text | Optional |
| `created_at` | DateTime | Required |
| `updated_at` | DateTime | Required |

Allowed `document_type` values:

```text
invoice
receipt
report
bank_document
other
```

Allowed `status` values:

```text
new
in_progress
completed
missing_information
```

Rules:

- `vat_amount` and `total_amount` are recomputed server-side from `amount_before_vat` and `vat_rate` unless a clearly defined reverse-calculation input is used.
- Do not trust client-provided file paths, MIME types, calculated totals, or stored filenames.
- The original file name is for display only; never use it directly as the storage path.
- The API must not reveal `file_path` to the frontend if it can expose server implementation details.

Suggested indexes:

- `client_id`
- `(client_id, document_date)`
- `status`
- `document_type`
- `document_date`

### 4.6 `payments`

Represents manual internal payment records only.

| Field | Type | Rules |
|---|---|---|
| `id` | Integer | Primary key |
| `client_id` | Integer | Required foreign key to `clients.id` |
| `document_id` | Integer | Optional foreign key to `documents.id` |
| `amount` | Numeric(14,2) | Required; must be `>= 0` |
| `status` | String(30) | Required enum |
| `payment_date` | Date | Optional |
| `notes` | Text | Optional |
| `created_at` | DateTime | Required |
| `updated_at` | DateTime | Required |

Allowed `status` values:

```text
unpaid
paid
partially_paid
pending
failed
```

Rules:

- No card number, CVV, bank credentials, payment-token, or checkout data may be stored.
- A later version may derive document payment status from payment records; the MVP may display the manual records directly.

Suggested indexes:

- `client_id`
- `document_id`
- `status`
- `payment_date`

| Field | Type | Rules |

Seed service names:


## 5. Relationship and Deletion Rules

- A document belongs to one client.
- A task belongs to one client and may belong to one document.
- A payment belongs to one client and may belong to one document.
- Do not silently cascade-delete documents, tasks, and payments when a client is deleted.
- Prefer archive/deactivation behavior for clients in the UI when records already exist.
- If permanent delete remains in the MVP API, block it with a clear validation error when dependent records exist, or require an explicit controlled deletion process.
- When a document is deleted, remove its private file only after database/authorization checks succeed; log/handle missing files safely.

## 6. Pydantic Schema Pattern

For every major entity, create separate schemas where appropriate:

```text
<Entity>Base
<Entity>Create
<Entity>Update
<Entity>Response
```

Rules:

- Do not include `password_hash` in any response schema.
- Do not include private absolute file paths in response schemas.
- Validate enum values, non-negative amounts, dates, maximum string length, and foreign-key references.
- Update schemas should have optional fields but must validate meaningful combinations.

## 7. Future Database Work — Not MVP

Do not implement these until explicitly approved:

- Alembic migrations.
- PostgreSQL configuration.
- Audit-log table.
- Multi-office tenancy.
- Full fine-grained permission tables.
- Soft-delete implementation across all tables.
- Client portal tables.
- External integration credentials stored in database.

When production work begins, add migrations, backups, audit logging, retention policies, and a proper PostgreSQL deployment plan.
